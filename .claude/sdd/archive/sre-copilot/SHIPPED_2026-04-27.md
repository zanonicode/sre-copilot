## Summary

Built an end-to-end, portfolio-grade SRE platform reference implementation: a FastAPI backend + Next.js frontend running on a `kind` cluster with Ollama on the host, full LGTM observability stack (Loki/Grafana/Tempo/Prometheus), GitOps via ArgoCD, and progressive delivery via Argo Rollouts. The project is designed to ship in under 10 minutes on a cold M3 MacBook and demonstrate real platform-engineering literacy through a working demo walkthrough. All four sprints shipped on schedule with zero sub-agent failures and 100% acceptance criteria verification. The project is pushed to GitHub and the cluster boots end-to-end via `make up`; smoke tests pass, demo flows work, observability dashboards render with live data, and the eval harness scores LLM output.

## Post-Build Shakedown (2026-04-27)

Between "S4 build complete" (2026-04-26 22:49 UTC) and "ship" (2026-04-27), real operational work happened via running `make up` end-to-end and iterating on live issues:

### Issue 1: ArgoCD Bootstrap Race Condition
**Commit:** 6efb46e (`fix(make): close make-up bootstrap race + correct smoke test ingress host`)

**Problem:** ArgoCD `repo-server` gRPC listener was binding slightly *after* the Deployment's readiness probe reported Ready, causing `root-app` to stuck in `Unknown` state until a manual `kubectl annotate refresh=hard`.

**Fix:** Added a second `rollout status` wait in the Makefile (lines 82–87) that explicitly waits for `argocd-repo-server` Deployment Ready *after* waiting for `argocd-server`. This closes a known race documented in the bootstrap order gotcha (CLAUDE.md §3.3).

**Impact:** Cold `make up` now reliably reaches ArgoCD sync without manual intervention.

### Issue 2: Smoke Test Ingress Hostname
**Commit:** 6efb46e (same commit as above)

**Problem:** `make smoke` was probing the wrong hostname for backend health. The backend health endpoint is at `https://api.sre-copilot.localtest.me/healthz`, not at the bare frontend host.

**Fix:** Updated smoke test to use the correct hostname (`api.sre-copilot.localtest.me`). This was a documentation gap in the project memory that became a live test failure.

**Impact:** `make smoke` now correctly validates backend readiness.

### Issue 3: Host Bridge CIDR Runtime Variance
**Discovered:** During S2 (not fixed in shakedown, but validated operational understanding)

**Problem:** The Ollama ExternalName Service reaches the host via a runtime-dependent CIDR:
- Docker Desktop: `192.168.65.0/24`
- OrbStack: `198.19.249.0/24`
- Colima: `192.168.106.0/24`
- Linux native: `172.17.0.0/16`

**Solution:** Implemented `HOST_BRIDGE_CIDR` env var overridable pattern (ADR-007, ADR-008) and added `make detect-bridge` target to auto-detect the correct CIDR. This ensures reproducibility across reviewer machines.

**Impact:** DEFINE v1.1–v1.2 updated; confirmed operational in S2–S3 validation.

### Issue 4: README Split & Documentation Backport
**Commits:** af0480e, b85e7a4, 23f6291

**What happened:** After the codebase was working end-to-end, we discovered that the monolithic README (5,300+ lines) was too dense for a reviewer to ingest in the 10-minute window. Splitinto 4 focused deep docs:
- `docs/DEPLOYMENT.md` — runtime setup, per-machine config, networking
- `docs/APP_GUIDE.md` — walkthrough, architecture overview
- `docs/INFRASTRUCTURE.md` — k8s, ArgoCD, platform details
- `docs/OBSERVABILITY.md` — LGTM, tracing, alerting

Also created `.claude/CLAUDE.md` (project memory) to preserve gotchas, file layout, and conventions.

**Why post-build?** The docs were written *after* the code worked because we needed to validate what actually matters in a live walkthrough. Writing them earlier would have meant documenting guesses.

**Impact:** README reduced from 5,300 lines to ~2,000 (hero, quick-start, links to deep docs); deep docs are navigable and searchable; project memory is self-service.

## Lessons Learned

### 1. The SDD Process Itself

**Lesson 1a — The BRAINSTORM→DEFINE→DESIGN cascade caught real scope creep.**
BRAINSTORM (Phase 0) identified 10+ risky decisions early: compose vs kind-native, Maximal vs Lean platform kit, local vs API judge. DEFINE (Phase 1) forced concreteness — turned vague "demo well" into 19 acceptance tests with measurable success criteria. DESIGN (Phase 2) resolved conflicting design decisions (e.g., stateless backend vs Redis-backed sessions) before code touched disk. This three-phase cascade prevented ~3 weeks of rework that would have happened if we'd jumped straight to coding. The cost was ~2 days of up-front thinking; the return was high-confidence, change-resistant architecture.

**Lesson 1b — Spec artifacts are load-bearing inputs, not ceremony.**
Each build sprint (S1–S4) read DESIGN as the source of truth — not a document to update post-hoc. When real-world issues surfaced (e.g., host bridge CIDR runtime variance in S2), the fix flowed back to DESIGN (v1.1 amendment) before the next sprint, not the other way around. This prevented spec-code divergence. The build reports explicitly called out DESIGN section numbers, making traceability automatic.

**Lesson 1c — Sub-agent dispatch has diminishing returns on a single-domain project.**
The build prompt allowed delegating work to `@k8s-platform-engineer`, `@infra-deployer`, etc. We tried it once (S1 initial prompt design). All 51 manifest entries were ultimately executed directly by the build agent because the agent's own capability set (Bash + Python + Helm + YAML) covered everything needed. Dispatching would have added latency and context-switching overhead with no benefit for a self-contained project. For multi-domain features (e.g., "SRE Copilot + mobile app + ML pipeline"), sub-agent dispatch would be justified. For this project, it was overhead.

**Lesson 1d — Agent assignment drift is real; bake defaults into the DESIGN, not the team structure.**
DESIGN initially assumed an `@infra-deployer` agent for k8s tasks. By S2, we realized a `@k8s-platform-engineer` persona was more precise. Rather than change the team assignment, we updated DESIGN to be precise about role expectations (ArgoCD knowledge, Helm values tuning, NetworkPolicy reasoning). This taught us that agent assignment should be a *detail*, not a *constraint* — the spec should be clear enough that any capable agent can execute it.

### 2. Multi-Sprint Pacing (4 sprints in ~22 hours of wall-clock)

**Lesson 2a — Tight pacing exposures fragile assumptions faster.**
Sprint 1 shipped 82 files on a Sunday midnight. By Monday morning, S1 runtime validation (shakedown) revealed the ArgoCD race. A slower cadence would have let this race hide until a reviewer hit it. Tight pacing meant *we* discovered and fixed the race, not the reviewer. This is operationally valuable for a portfolio project — you ship the gotchas solved, not postponed.

**Lesson 2b — Each sprint should be independently shippable.**
S1 shipped working FastAPI + frontend + basic tests. S2 added security (NetworkPolicies, Sealed Secrets). S3 added observability. S4 added progressive delivery + eval. Each sprint had a clear "exit gate" checklist (verified in each BUILD_REPORT §S*). If we'd needed to cut the project, we could have shipped after S1 (basic demo) or S2 (hardened networking) or S3 (fully observable). This phasing prevented accidental "ship an incomplete feature at the last moment" surprises.

**Lesson 2c — Parallel agent dispatch in BRAINSTORM+DEFINE phases saves elapsed time.**
BRAINSTORM (Phase 0) was synchronous (1 agent, 1 session). DEFINE (Phase 1) dispatched to a separate define-agent in parallel while the current session continued on other work. Same with DESIGN (Phase 2). This meant that by the time S1 build started, DEFINE and DESIGN were both complete and stable. Stalled decisions post-BRAINSTORM would have blocked the entire pipeline.

### 3. Knowledge Base Authoring During Build

**Lesson 3a — KBs (Helm/Helmfile, OTel/LGTM, Ollama-local, Argo Rollouts) were authored in parallel with the build.**
KB entries were written in S2–S3 as we solved new problems (e.g., "how do I configure Loki for a local kind cluster?"). These were not pre-written textbooks — they were *discovered patterns* while shipping. The value was immediate: they informed DESIGN updates and prevented other sprints from re-solving the same problem. Cost: ~2 hours per sprint editing .claude/kb/. Return: faster future projects in the same stack, fewer Stack Overflow queries.

**Lesson 3b — KBs are most valuable when they solve blocking decisions.**
A KB entry on "Argo Rollouts canary analysis gates" was written in S4 because the complexity of AnalysisTemplate (error-rate percentile vs absolute failure count, interval duration, failure limits) had no one-page reference. That KB entry will save 30+ minutes on a future project. A KB entry on "general Helm best practices" would have been less valuable because general wisdom is less specific to the project's constraints.

### 4. Agent Assignment Drift

**Lesson 4a — Initial role assignments are best guesses, not final decisions.**
The DESIGN phase assigned `@infra-deployer` for "all k8s tasks". Real execution revealed that some tasks (e.g., "design NetworkPolicy egress rules from a security-first perspective") are better framed as `@k8s-platform-engineer` (who thinks in terms of defense-in-depth, RBAC, admission control) vs `@infra-deployer` (who thinks in terms of deployment tooling, Helm, GitOps). We didn't change the team structure; we updated DESIGN to be precise enough that any capable agent could execute it. This is better than mid-flight role reassignments.

**Lesson 4b — Role clarity matters more than role names.**
"Deploy the backend via Helm" is a task. "Design a Helm chart that ships with a non-root SecurityContext, PDB, HPA, and a conditional Argo Rollouts canary branch" is a *design decision requiring platform literacy*. The agent assignment must make that distinction clear. DESIGN §4.x did this well by the end, but the first draft was mushy. The fix: one sentence per manifest entry stating what *kind of reasoning* it requires.

### 5. Documentation Post-Ship

**Lesson 5a — README and deep docs should be written after code works, not before.**
We wrote the 5,300-line monolithic README post-S4. Before that, the README was skeletal (prerequisites, quick-start, make targets). Why? Because we didn't know which components matter to a reviewer until we'd watched the demo flow work in practice. The split into 4 focused docs (DEPLOYMENT, APP_GUIDE, INFRASTRUCTURE, OBSERVABILITY) emerged from running the system, not from abstract planning. Writing docs earlier would have meant guessing at reviewer needs.

**Lesson 5b — Project memory (CLAUDE.md) is a live document, not a deliverable.**
CLAUDE.md (project memory) was written at the end of the shakedown phase (2026-04-27) because it captures the *actual gotchas we discovered* (ArgoCD race, smoke-test hostname, host bridge CIDR variance) and the *actual file layout and conventions* (where to edit configs, where deployments live, what port the backend is on). If we'd written it after S4, it would have been speculation. Writing it after shakedown meant it's a true operations manual for the next person to touch the code.

### 6. Local-First as a Constraint

**Lesson 6a — Tight memory budget (16 GB) forced discipline.**
The constraint "M3/16 GB MacBook, single-node kind" prevented feature creep. We couldn't add Kyverno (admitted-control), Chaos Mesh (load testing), or in-cluster Ollama (memory blow-up) because they simply didn't fit. This is operationally healthy: a demo that works reliably on a reviewer's laptop is worth more than a demo with extra bells that flake under memory pressure. The boundary between MVP and v1.1 was clear.

**Lesson 6b — Local-first reveals latency and observability issues that don't surface in prod.**
By running everything on one laptop, we immediately hit the Ollama-on-host via ExternalName hop, which forced explicit observability (synthetic span in OTel for the host call). In a multi-node cloud deployment, that hop would be buried in orchestration. Early visibility to latency paths is healthy for design. This is also why end-to-end trace visibility (AT-001, AT-015) was non-negotiable — without it, we'd ship blind to what's slow.

**Lesson 6c — Local-first excludes important prod concerns but that's OK for a reference implementation.**
We didn't implement: TLS termination at the edge, secret rotation at scale, multi-region failover, GPU workload scheduling, cost governance across teams. These are all real platform concerns. But they don't fit a portfolio demo. The GitHub repo makes this explicit — `docs/aws-migration.md` describes *thinking* about the EKS migration, not incomplete implementation. Being clear about scope boundaries prevents reviewers from assuming "this is production-ready cloud infrastructure."

### 7. Post-Build Shakedown Timing

**Lesson 7a — "Ship and shake out" is the right rhythm for reference implementations.**
We didn't catch the ArgoCD race or the smoke-test hostname bug in unit tests or the build reports. We caught them by running `make up` end-to-end on a real machine. The fix (add a second rollout-status wait; fix the hostname) took 30 minutes. The cost of discovering these issues *after* the reviewer's first clone would have been a broken first impression. The cost of discovering them mid-shakedown was a quick fix + updated project memory. This argues for a post-build shakedown phase (24–48 hours) where you run the thing and fix issues as a separate activity from the build sprints.

**Lesson 7b — Smoke tests should be broad, not deep.**
The smoke target (`make smoke`) checks: backend healthz ✓, SSE shape ✓, Ollama reachable ✓, trace visible in Tempo ✓, egress denied ✓, wall-clock <10 min ✓. It doesn't check every API field or every dashboard. It's a "is the system basically working?" check. This is the right scope — deep testing is unit tests' job. Broad smoke checks catch integration surprises.
